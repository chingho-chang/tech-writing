# 先把話說清楚，剩下的交給 agent

*一週 spec-driven 的 agentic coding，如何讓文件和測試自然長出來*

---

> ### *Find purpose, the means will follow.*

---

一整天，一個人寫了 spec、寫了設計文件、補了測試、歸檔了好幾個變更。最反常的不是速度——而是那些向來會往後拖的事：寫文件、補測試，這次居然順手就做完了。

過去很多工程師嘴上說該寫文件、該寫測試，手上永遠先做功能。文件留到「之後補」，測試留到「有空再說」，然後就沒有然後了。但這一週用 Claude Code + Opus 4.7 搭 Spectra（一套 spec-driven 的 openspec 工作流）做一個業餘專案，我注意到一件事悄悄變了，而且**不是因為突然變自律**。

這篇想講的不是「AI 讓寫程式變快」——那是大家都在講的事。我想講一個更反直覺、對技術管理更有意思的觀察：

> AI 協作真正改變的，不是寫程式的速度，而是把軟體工程裡兩件長期靠紀律維繫的苦差事——**寫文件、寫測試**——從「美德稅」變成了「自然發生的副產品」。

---

## 一、兩個慢性病：軟體工程的美德稅

先說清楚所謂的「美德稅」是什麼。

文件和測試有個共同的尷尬處境：**它們對當下是純成本，對未來才有益**。你今天花兩小時寫設計文件，今天的功能一行都沒多；你今天花一小時補端對端測試，今天的展示一樣能跑。好處要等到三個月後別人接手、或某次重構半夜炸鍋時才兌現——而且兌現的人常常不是當初付出成本的你。

於是它變成一種「美德」：做得越多越值得稱讚。但真正的戰場其實不在「做或不做」——現在的流程不太允許你完全不做，CI、覆蓋率門檻、review 清單都守在那裡。真正會發生的是：大家都做到「剛好過關」就收手，補幾條快樂路徑的測試、寫兩行交差的說明，沒人擋得了合併上線，而每個人心裡都清楚，這離「足夠」還很遠。這些制度頂得住底線，卻頂不住那條線之上的一切——期限一來，「做多少」就一路往最低點滑。本質上，它們都是在用**紀律**對抗**誘因**。

但凡要靠紀律維繫的東西，規模一大就會漏。這不是工程師懶或不專業，這是誘因結構的問題：當「先把功能做出來」的即時回報遠大於「寫好文件測試」的延遲回報，理性的人就會系統性地拖欠這兩筆稅。

一直以為這是軟體工程的結構性宿命。直到這一週才發現：原來 AI 協作把這個誘因結構整個翻過來了。

---

## 二、工作流現場：一個人怎麼跑完一個完整 SDLC

這是一個有 LLM 參與的個人 web app：後端 Hono + better-sqlite3 純本機跑，前端一般的 SPA，核心是把結構化輸入丟給 Anthropic API、拿回結構化輸出再持久化。開發環境是 Claude Code（跑 Opus 4.7），搭一套叫 Spectra 的 spec-driven 工作流——你可以想成 openspec 那一類：每個改動都是一個 **change**，走 `discuss → propose → apply ⇄ ingest → archive` 的生命週期。

> openspec 是一種把 spec 放到開發流程正中央的做法——每個改動先寫提案、spec 與任務清單，再動手實作；Spectra 是這套做法的一個工具實作，把 discuss、propose、apply、archive 這些階段包成可直接操作的指令。

實際跑起來像這樣：

- **discuss**：先把要做的事跟 AI 對話釐清，確認幾個小議題的方向（要不要新狀態、邊界怎麼定、測試走哪條 path）。需求清楚就跳過。
- **propose**：AI 生出一份 proposal、一份 design.md、若干 spec delta、一份 tasks 清單。這是「合約」階段。
- **apply**：照 tasks 一條條實作。一個 change 動輒 14–21 個 task，跨前後端加測試。
- **ingest**：實作到一半發現設計要改？不是默默改 code，而是回頭把 design / spec / tasks 三處同步更新，再繼續。
- **archive**：跑完、測過、把 spec delta 併回正式 spec，收進 archive。

值得一提的是，這整套狀態不是只能埋在密密麻麻的命令列輸出裡——Spectra 有一個視覺化介面，把每個變更走到哪一階段、每條任務勾了沒，都攤在眼前。對一個人同時 hold 住多個並行變更來說，「一眼看到全局」比想像中重要：你不會忘記某個變更卡在 ingest 還沒收尾，也不會漏掉哪條任務還沒跑。

一週下來跑了 8 個這樣的 change，光一天就走完數個完整週期。其中一個 change 的核心 code 改動只有 6 行（替 API 呼叫加上 cache 設定），但它一樣老老實實走完 `propose → design → tasks → apply → 測試矩陣 → archive`。

這裡有個關鍵心態轉變：**不再把 change 當成「功能粒度」，而是「一次內聚的決策粒度」**。哪怕只改 6 行，它都值得一條可追溯的決策軌跡。過去這種「小改動還要寫文件喔？」的抗拒感，現在完全消失了——因為寫文件的成本已經不在我身上了。這就帶到第一個翻轉。

## 三、翻轉一：spec 不再是「事後文件」，而是「給 AI 的前置合約」

過去寫 spec 最大的心理障礙，不只是它的好處要等很久才兌現，更在於——**你在寫的當下，根本無法確認它未來到底有沒有用**。spec 的讀者是三個月後的某個人，可能是同事，更可能是早就忘光脈絡的自己。問題是你只能用「此刻、脈絡滿載的自己」去寫；那些此刻覺得「這還用寫嗎」的理所當然，往往正是三個月後讀不懂的盲點——而你當下無從判斷哪些該留、哪些是廢話。回報既遲、又不確定，spec 在誘因天平上自然更輕。

但在跟 AI 協作時，spec 的讀者變了——**spec 的第一讀者是 AI，而且是現在**。

這個轉變的力道超乎預期。當你不寫清楚 spec，AI 不會「將就做」，它會**漂移**：自己腦補需求、幻覺出你沒要的欄位、往錯誤的方向一路狂奔 200 行。我這週吃過好幾次虧——口頭講個大概就讓它 apply，結果做出一個「結構上很完整、但完全不是我要的」東西。spec 沒寫清楚的代價，不是三個月後償還，而是**五分鐘後立刻當著你的面爆出來**。

於是寫 spec 的誘因被徹底翻轉：它從「該做但可以拖」變成「不寫就動不了」。延遲回報變成了即時回報。

更關鍵的是另一半：**書寫成本趨近於零**。我不用自己從白紙寫 design doc，只跟 AI 對話釐清想法，它把口語討論轉成結構化的 proposal / design / spec / tasks，我只需要 review 和反覆討論修改。寫文件這件事，從「一個人對著空白編輯器硬擠」變成「審一份草稿」——心理門檻天差地別。

這一週還具體學到兩件關於「怎麼寫好給 AI 的 spec」的事：

- **few-shot 範例遠勝抽象規則**。一開始想用「語氣要溫暖、不要傲慢」這種抽象 rule 去約束輸出，反覆失效。改成直接給 4–6 個正反範例後，行為立刻穩定。AI 對「示範」比對「說教」敏感得多。
- **規則的組織方式是維護性的分水嶺**。我的 system prompt 一開始是 14 條扁平編號的規則，加第 15 條時得 cascade 改掉一堆 cross-reference。後來重組成主題分組（§A 結構、§B 時間、§C 完整性…），新增規則只 append 不破壞既有引用。spec 跟 code 一樣會長技術債，而且同樣需要重構。

行，文件的問題 AI 幫忙解了，那測試呢？測試甚至更討厭。

## 四、翻轉二：測試不再是「品質美德」，而是「敢放手讓 AI 改的安全網」

測試的美德稅比文件更難收。文件至少寫一次就有，測試是你每改一次 code 都得維護的持續成本。

但 AI 協作在這裡也翻轉了誘因，而且翻轉的理由很硬：**AI 一次改動的範圍，大到逐行讀完不再實際**。

一個 change 動 17 個檔案、+1070 −35 行是這週的日常。沒辦法像 review 同事 PR 那樣逐行看完還有信心——但這逼著想清楚一個一直被混為一談的問題：**逐行 review，到底在確認什麼？**

其實是兩件事。一件是**行為與契約**：這系統做什麼、介面和邊界對不對、失敗怎麼處理。另一件是**實作細節**：這個迴圈怎麼寫、變數叫什麼、用了哪個 helper。過去它們綁在一起，是因為確認行為的唯一手段就是讀實作。測試把它們拆開了——spec 鎖住「該做什麼」、測試鎖住「不准做錯什麼」，於是能在**行為層保持完全掌握**，把**實作層**放給 AI。

所以放手的從來不是「掌握」，而是「逐行讀實作」這個低效動作。契約、邊界、還有測試難覆蓋的高風險區（安全邊界、資料遷移、刪除性操作），反而看得比以前更仔細。測試不是用來「代替」我審查，而是把注意力從低價值的逐行閱讀，**重新分配**到真正該盯的地方——它從「品質保證的加分項」，變成了讓這個重新分配得以成立的槓桿。

而且——測試也是 AI 幫我寫的。這週我建了兩層：

- **curl matrix**：針對 LLM 輸出的回歸矩陣，一次 34 個 case（baseline / 各種邊界 / 各種模式），五分鐘內 burst 跑完，自動驗每個 case 的輸出符不符合 invariant。改完 prompt 立刻跑一輪，抓「無聲的退化」——那種不會報錯、但行為悄悄變掉的 regression。我甚至故意改壞一個 function 驗證矩陣抓不抓得到，確認安全網是真的。
- **Playwright e2e**：跑真實 Chromium + 一個獨立測試 DB + STUB_LLM（把 LLM 換成固定 fixture，讓測試 deterministic）。我寫了 4 條 flow spec 覆蓋主路徑和幾個分支。這層抓到好幾個 component test 看不到的 bug——跨頁狀態、資料流經過真實 proxy 才會炸的問題。

注意這兩層的成本結構：**setup 一次，之後每次改動的邊際驗證成本趨近於零**。這正是過去測試最被詬病的地方（維護成本高）被 AI 抹平了——fixture、assertion、新 spec 都是 AI 生的，我改 code 它順手把測試也改了。

於是「該不該補測試」這個糾結消失了。不是因為變勤勞，是因為**不補測試我就不敢按 merge**，而補測試又幾乎不花什麼時間。誘因和成本同時被翻轉，紀律就不再是必需品。

## 五、誠實面：成本沒有消失，它換了科目

如果文章到這裡結束，那就成了一篇工商。這套東西有它扎實的代價，而且對要做採用決策的人來說，關鍵不在「成本少了多少」——而在**成本換了科目**：你省下的是每天都要繳的「營運稅」（靠紀律硬擠出來的文件和測試），新增的卻是一次性的「資本投資」（驗證 AI 的基礎設施）。淨值為正的真正原因是這個科目轉換，不是總量下降。

這週付了兩筆這樣的投資，講清楚它們長什麼樣、為什麼是一次性的。

**成本科目一：驗證 AI。** LLM 最危險的地方不是會出錯，而是**它不丟 exception，會給你一個「結構合法、語意違規」的輸出**。我這週遇過它把一個該是特定數值的欄位算成別的、甚至生出 `21:60` 這種根本不合法的時間——格式完全正確，校驗稍不留神就放它過去——AI 最棘手的地方就在這裡：它最擅長產出「看起來對」的東西。

但重點是：**這個風險是已知、可控、可自動化的**，不是黑天鵝。對策是把關鍵規則從 prompt 裡的「請你遵守」升級成 server 端的硬斷言——跑一組 `validatePostConditions`，違反就帶著「你違反了哪一條」自動重新呼叫 LLM，最多 retry 三次，三次都失敗才降級。背後是一個對 LLM 應用的核心認知：**prompt 是 best-effort instruction，不是 contract**；它對規則的服從度是機率分布，再清晰的措辭也只降低違規率、不會歸零。能機械判定對錯的硬約束，不能只靠 prompt 請 LLM 遵守——得用同一份 spec，在程式碼裡架一道驗證關卡。prompt 仍然承載大量軟性的判斷與風格，它只是不能當「保證」。

從**重工**的角度看，這層投資的回報最直接：沒有它，AI 的靜默錯誤會一路流到下游，等你發現時已經要重做；有它，錯誤在自動關卡就被擋回重試——**連人都不用介入，重工根本沒發生**。

**成本科目二：迭代流量。** 某天下午那個 34-case 的回歸矩陣一跑，整批 HTTP 529——撞上了 Anthropic 的 ITPM（每分鐘輸入 token）限制。一開始以為是 API 偶發不穩，加了 sleep 還是撞。後來才想通：這不是偶發，是 **agentic dev loop 的結構性特徵**——每次改 prompt 都要 burst 跑一輪矩陣，這個流量尖峰跟「prompt 穩不穩定」毫無關係，是開發節奏本身固有的。

解法是啟用 prompt cache：把靜態的 system prompt 和 tool 定義標上 `cache_control: ephemeral`，核心改動 6 行。效果立竿見影——同樣的 burst 測試，Uncached token 尖峰從撞牆的 85K/min 降到約 12K，cache 命中率從 0 跳到約 90%，12 連發 38 秒內 0 失敗。值得記下的判斷錯誤是：我原本想「等 prompt 穩定再來加 cache」，這是錯的——真正的成本來源是 dev loop 的 burst，跟 prompt 穩不穩定無關。**cache 不是事後優化，是 agentic 開發的 day-1 基礎設施。**

把帳算清楚，給要推這套方法的人一個總結：

- **成本**：是一次性的工程建設（驗證層、cache），不是持續的人力投入；建好之後邊際成本趨近於零。
- **風險**：新風險（LLM 靜默失效）是已知、可控、可自動化的，不是不可預測的黑天鵝。
- **重工**：最大的紅利在這裡——漂移在 spec 階段就攔、靜默錯誤在驗證關卡就擋，重工從「做完才發現」前移成「進門就被擋下」。

所以這不是免費午餐，但它是一筆**划算的科目轉換**：把每天都要重新對抗一次的人性，換成一次建好、之後自動運轉的基礎設施。

退一步看，剛剛那道程式碼驗證關卡，跟前面講的 spec、測試其實是同一件事的三個面向。一份 spec 寫清楚之後，它會同時長出三隻手：prompt 照它生成、程式碼 validator 照它把關、測試照它回歸驗證——**一份 spec，三種投影**。這正是為什麼「文件、可靠性、測試」這三件過去各自拖欠的事，如今會「一起」到位：它們本來就源自同一個定義。你真正要花力氣的，從頭到尾只有一件——把那份 spec 想清楚。

## 六、收尾：誘因重構，而不是紀律加碼

如果這一週只能留下一個體會，那會是：**真正要花力氣的，從頭到尾只有把 spec 想清楚。** 其他的——prompt 照它生成、測試照它驗證、AI 照它實作——都是這份定義的下游。

這件事悄悄改寫了「工程師在做什麼」，花在「逐行寫實作、逐行讀 diff」的時間大幅下降，花在「把需求想清楚、把邊界定義好、守住高風險區」的時間大幅上升。手沒有變閒，但**力氣從實作層，重新分配到了定義層**。而文件、測試、可靠性這些過去被歸為「該做但沒空做」的事，剛好全都長在定義層上——於是它們不再是額外負擔，而是我本來就在做的那件事的副產品。

回頭看開頭那句反常的觀察：開始寫文件和測試，不是因為變自律，而是因為**它們的誘因結構被翻轉了**。spec 從「給未來的人」變成「給現在的 AI」，不寫就動不了；測試從「品質美德」變成「敢放手的前提」，不寫不敢放手。還記得開頭那條「剛好過關就收手」的滑尺嗎？當寫清楚的即時回報變明確、書寫成本又趨近於零，那條滑尺就自然往上滑了——不是被紀律逼上去的，是被誘因拉上去的。

這對技術管理的啟示，比「AI 寫 code 很快」重要得多：

> 過去我們想提升團隊的文件和測試，手段幾乎都是**加碼紀律**——更嚴的 review、更死的 CI gate、更多的流程。但凡要靠紀律維繫的，規模一大就漏。AI 協作給了另一條路：**重構誘因**。值得投資的不再是「逼大家交稅的流程」，而是「讓團隊把 spec 想清楚的能力」，加上那套一次建好、之後自動運轉的驗證基礎設施。當寫 spec 是「讓 AI 不漂移的前提」、寫測試是「敢放手的前提」，這兩件事就不必再逼，它們會自己發生。

一個人 + AI 能跑完一個完整 SDLC，最有意思的地方從來不是「取代了一個團隊」。而是當人的力氣被重新分配到「把事情定義清楚」上，那些過去該做、卻總是排在功能後面的事——文件、測試、可靠性——第一次和功能本身，變成了同一件事。

## 附錄：加速、放大、自動導航——這週的指令與 skill

光講流程是空談，這裡補上實際解決問題的操作。

**Spectra 流程骨架**：`discuss →（需求清楚可跳過）propose → apply ⇄ ingest → archive`。前段把需求收斂成 spec，中段照任務實作，需求一變就用 ingest 回頭同步文件，最後 archive 把 spec 併回主線。整條骨架就是一套繞著 spec 打造的自動導航——spec 標出目的地，每個階段一路加速推進，把一個人的力氣放大成一支團隊的產出。

### Spectra 流程指令

- **`/spectra-discuss`** — 痛點：需求模糊就開工，AI 會把你沒講清楚的地方自己腦補，做出一個「完整但不對」的東西。解法：開工前先把模糊需求拆成 2–3 個可以當場拍板的小議題，逐一定案，再進 propose。
- **`/spectra-propose`** — 痛點：自己對著空白編輯器寫設計文件，門檻高到讓人想跳過。解法：一次生出提案、設計、spec 差異與任務清單，我只需要 review 和改——書寫成本從「從零產出」降成「審一份草稿」。
- **`/spectra-apply`** — 痛點：一個變更動輒十幾個檔案，邊做邊改範圍、漏步驟，很容易失控。解法：照任務清單逐條實作、完成一條勾一條，範圍要變就轉 ingest——讓「實作」本身也可追蹤、可中斷續做。
- **`/spectra-ingest`** — 痛點：實作到一半發現某個設計決策要調整，最自然的衝動是默默改 code，文件就此 drift。解法：把變更先回灌進設計、spec、任務三處同步，再繼續實作。這一步是「文件永遠對得上 code」的真正關鍵。
- **`/spectra-archive`** — 痛點：每個變更各自帶一份 spec 差異，散著放久了就沒人知道現況。解法：完工時把差異併回正式 spec，維持單一真相源。
- **`/spectra-commit`** — 痛點：多個變更並行時，`git add` 很容易掃進不相關的檔案。解法：只提交屬於這個變更的檔案，保持提交歷史乾淨可追溯。

（其餘像 `/spectra-ask` 查 spec、`/spectra-verify` 驗實作對不對得上 spec，按需取用。）

### 各階段的護欄：superpowers skill 怎麼盯著 agent 不離開 spec

Spectra 給的是「流程骨架」——該走哪些階段。但真正讓 agent 在每個階段都緊扣 spec、不自由發揮的，是一批 superpowers skill 在旁邊輔助與監督。它們不是獨立工具，而是嵌進各階段的護欄：

- **discuss 階段 ─ `brainstorming`**：在需求還沒寫成 spec 之前，逼著把意圖、限制、成功標準一個個問清楚，並逼出 2–3 個取捨方案。它擋住的是「跳過釐清就衝進實作」——確保進到 spec 的方向本身是對的。（這篇文章就是這樣一問一答逼出來的。）
- **propose 階段 ─ `writing-plans`**：把 spec 拆成有順序、可驗收的步驟，agent 照計畫推進而非即興發揮；計畫本身也成了一道「對不對得上 spec」的檢查清單。
- **apply 階段 ─ `verification-before-completion`**：實作完不等於完成。它要求在宣稱「做好了」之前先拿出證據（我甚至故意改壞一個 function，確認測試真的會擋）——擋住的正是 LLM 最擅長的「看起來對」。
- **跨階段・出狀況時 ─ `systematic-debugging`**：當 LLM 行為跑偏，不是讓 agent 東改西改碰運氣，而是系統化定位失敗模式再下手；我那串 LLM 可靠性的修補就是這樣逼出來的。

一句話：**Spectra 標出航線、推著開發一站站前進；superpowers skill 則在沿途自動修正航向，agent 一偏離 spec 就把它拉回來。**

### Last but not least — 版本控制：讓試錯成本變低的，是隔離

agentic 流程裡，真正把試錯成本壓低的，不是「搞砸了再回滾」，而是 **AI 從頭到尾都在一條隔離的線上跑**。superpowers 的 `using-git-worktrees` 做的正是這件事：開工前先把工作落到獨立的 branch 或 worktree，主線完全不暴露在風險裡。

於是 AI 可以在那條隔離線上放膽改、放膽試，方向不對就整條丟掉，主線毫髮無傷；真正要併回主線的，只有通過測試把關的那一份（這道收尾的關卡就是 `finishing-a-development-branch`）。試錯的成本與風險之所以低，不是因為「回得來」，而是因為主線**根本沒被碰過**——這正好和翻轉二同調：測試讓你敢放手行為層，隔離讓你敢放手實作層。
